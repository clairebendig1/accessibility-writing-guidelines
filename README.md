# Accessibility Writing Guidelines for Technical Documentation

Writing guidelines for creating technical documentation that works for all readers — including those using assistive technologies, those with cognitive or learning differences, and those reading in non-native languages.

These guidelines are adapted from the accessibility writing standards I authored for developer documentation at Meta's Reality Labs.

## Why this matters for technical documentation

Accessibility in documentation is often treated as an afterthought — alt text gets added at the end, plain language is considered "dumbing down," and heading structure is treated as aesthetic. These guidelines treat accessibility as a core quality dimension, the same way accuracy and completeness are.

Accessible technical documentation is clearer for everyone, not just for readers with disabilities. The techniques that help a screen reader user navigate a reference doc also help a sighted reader skim it. Plain language serves both non-native speakers and readers under cognitive load. Good alt text helps when images don't load.

---

## Scope

These guidelines apply to:
- Developer documentation (API references, SDK guides, tutorials)
- Platform documentation (integration guides, architecture docs)
- Contributor-facing documentation (style guides, process docs)

They do not cover UI accessibility (WCAG compliance in interfaces), only written content.
